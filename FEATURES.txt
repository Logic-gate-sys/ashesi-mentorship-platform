1. CORE SYSTEM ARCHITECTURE
================================================================================

SYSTEM OVERVIEW:
The Mentor App facilitates structured mentorship connections between students
and alumni through mentorship cycle management. The system handles user
registration, mentor discovery, mentorship request workflows, session
scheduling, real-time messaging, and admin oversight.

CORE ENTITIES:

User (Base Authentication Model)
├─ id (UUID)
├─ email (unique)
├─ password (hashed)
├─ role (STUDENT | ALUMNI | ADMIN)
├─ displayName
├─ avatar (optional image URL)
├─ createdAt
└─ updatedAt

MentorshipCycle (Temporal Container)
├─ id
├─ year
├─ season (SPRING | SUMMER | FALL | WINTER)
├─ status (PLANNING | ACTIVE | COMPLETED | INACTIVE)
├─ registrationOpenDate
├─ registrationCloseDate
├─ startDate
├─ endDate
└─ description

UserProfile (Role-Specific Details)
├─ StudentProfile
│  ├─ userId
│  ├─ yearGroup
│  ├─ major
│  ├─ interests []
│  ├─ goals (text)
│  └─ school
│
└─ AlumniProfile
   ├─ userId
   ├─ graduationYear
   ├─ company
   ├─ industry
   ├─ expertise []
   ├─ bio
   ├─ availability (weekly time slots)
   ├─ maxMentees
   ├─ currentMenteeCount
   └─ rating (0-5)

MentorshipRequest (Connection Request)
├─ id
├─ studentId
├─ mentorId
├─ cycleId
├─ status (PENDING | ACCEPTED | DECLINED)
├─ message (student's request message)
├─ respondedAt (when mentor answered)
└─ responseMessage

Session (Mentorship Meeting)
├─ id
├─ requestId
├─ scheduledDate
├─ scheduledTime
├─ duration (minutes)
├─ type (IN_PERSON | VIRTUAL)
├─ status (SCHEDULED | ONGOING | COMPLETED | CANCELLED)
├─ notes
├─ recordingUrl (optional)
└─ conversationId (linked chat thread)

SessionFeedback (Post-Session Feedback)
├─ id
├─ sessionId
├─ fromUserId
├─ rating (1-5)
├─ comment
└─ focusAreas []

Conversation (Chat Thread)
├─ id
├─ participants[] (studentId, mentorId)
├─ linkedSessionId (optional)
├─ createdAt
└─ messages[]

Message (Chat Message)
├─ id
├─ conversationId
├─ authorId
├─ content (text or markdown)
├─ attachments[] (optional files)
├─ sentAt
├─ readAt (optional)
└─ reactions (emoji)

Notification (System Alert)
├─ id
├─ userId
├─ type (REQUEST | REPLY | SESSION_INVITE | SESSION_REMINDER | MESSAGE)
├─ title
├─ message
├─ relatedEntityId
├─ isRead
├─ createdAt
└─ actionUrl

Availability (Mentor Weekly Schedule)
├─ id
├─ mentorId
├─ dayOfWeek (0-6)
├─ startTime (24h format)
├─ endTime (24h format)
└─ maxSessionsPerWeek


KEY ARCHITECTURE PRINCIPLES:
- Cycles create time-bounded cohorts (prevents mixing old/new mentors)
- Each mentorship request creates a one-to-one connection
- Sessions are discrete meetings within accepted mentorships
- Conversations can exist independently or link to sessions
- Availability is weekly recurring (simplifies scheduling)
- All state transitions trigger notifications

2. AUTHENTICATION & USER MANAGEMENT
================================================================================

REGISTRATION FLOW:
User selects role → Enters email/password → Creates profile based on role
├─ STUDENT: year, major, interests, school
└─ ALUMNI: graduation year, company, industry, expertise, bio

PROFILE SETUP:
├─ Student Profile
│  ├─ Interests (multi-select tags)
│  ├─ Goals (text field for learning objectives)
│  ├─ School (text field)
│  └─ Year (dropdown/selector)
│
└─ Alumni Profile
   ├─ Bio (rich text)
   ├─ Expertise (multi-select tags)
   ├─ Industry (dropdown)
   ├─ Company (text field)
   ├─ Graduation Year (selector)
   ├─ Weekly Availability (calendar interface)
   ├─ Max Mentees (number selector)
   └─ Profile Picture (upload)

AUTHENTICATION METHODS:
- Email + Password (primary)
- JWT tokens (session management)
- Role-based access control (middleware)

SESSION MANAGEMENT:
- Token stored in HttpOnly cookie
- Automatic refresh token rotation
- 24-hour expiration default

3. STUDENT FEATURES (Seekers of Mentorship)
================================================================================

STUDENT FEATURE 1: DISCOVER MENTORS
Purpose: Browse and search available alumni mentors
User Flow:
1. Student navigates to Mentors/Discovery page
2. Sees filtered mentor list with avatars, names, titles, ratings
3. Filters available:
   ├─ Industry (tech, finance, healthcare, etc.)
   ├─ Skills (matching their interests)
   ├─ Availability (next 7 days)
   ├─ Rating (4+ stars)
   └─ Company
4. Clicks on mentor card → Expands profile view with:
   ├─ Bio
   ├─ Expertise
   ├─ Experience level
   ├─ Mentee testimonials/ratings
   ├─ Calendar showing availability
   └─ "Send Request" button

Technical Details:
- GET /api/mentors?industry=tech&skills=python&availability=true&cycle=current
- Response includes: user info, profile details, ratings, availability
- Frontend caches mentor list (refresh every 30 min)
- Optimistic filtering (no server round-trip for client-side filters)

STUDENT FEATURE 2: SEND MENTORSHIP REQUEST
Purpose: Express interest in being mentored by specific alumni
User Flow:
1. After viewing mentor profile, student clicks "Send Request"
2. Modal opens with:
   ├─ Message field (optional, 500 char max)
   ├─ Goals checkbox (which goals align with this mentor's expertise)
   ├─ Accept ToS checkbox
   └─ Submit/Cancel buttons
3. Submit sends request to mentor
4. Success notification shown

Technical Details:
- POST /api/mentorship-requests
  payload: { studentId, mentorId, cycleId, message }
- Mentor receives notification immediately
- Request status: PENDING
- Student can send max 5 requests per cycle

STUDENT FEATURE 3: VIEW REQUEST STATUS
Purpose: Track sent mentorship requests and responses
User Flow:
1. Student navigates to "My Requests" page (under pending requests)
2. Sees list of requests with:
   ├─ Mentor name & photo
   ├─ Status badge (Pending, Accepted, Declined)
   ├─ Time since sent
   ├─ Response message (if declined)
   └─ For accepted: "View Sessions" link
3. Can withdraw pending request via context menu

Technical Details:
- GET /api/mentorship-requests?userId=student_id&status=pending|accepted|declined
- Real-time updates via WebSocket when mentor responds
- Webhooks trigger notification push

STUDENT FEATURE 4: SCHEDULE SESSIONS WITH MENTOR
Purpose: Book mentorship meetings within accepted mentorships
User Flow:
1. Student clicks "Request Session" from accepted mentorship
2. Calendar interface shows mentor's available time slots:
   ├─ Green blocks = available hours (from Availability schema)
   ├─ Red blocks = unavailable
   └─ Already booked times grayed out
3. Student selects timeslot and specifies:
   ├─ Session type (virtual or in-person)
   ├─ Duration (30/45/60 min options)
   ├─ Topic/agenda (optional)
4. Session created and both parties notified

Technical Details:
- POST /api/sessions
  payload: { requestId, type, scheduledDate, scheduledTime, duration, notes }
- GET /api/mentor-availability?mentorId&month=2025-03
- Response includes booked slots to prevent double-booking
- Calendar integrations (Google Calendar, Outlook) optional

STUDENT FEATURE 5: ATTEND SESSIONS & PROVIDE FEEDBACK
Purpose: Attend scheduled mentorship meetings and rate experience
User Flow:
1. Student receives reminder notification 24h before session
2. On session date, "Start Session" button appears (30 min before)
3. For virtual sessions: Joins video call (Zoom/Meet embedded)
4. Session timer shows duration remaining
5. After session ends, feedback form appears:
   ├─ Rating (1-5 stars)
   ├─ Comment field (optional, 1000 char)
   ├─ Focus areas (multi-select: communication, advice quality, etc.)
6. Feedback submitted and visible to mentor

Technical Details:
- PATCH /api/sessions/:id { status: ONGOING }
- Video integration: embedded Meet/Zoom iframe, or external link
- POST /api/session-feedback { sessionId, rating, comment, focusAreas }
- Ratings aggregate to mentor profile rating

STUDENT FEATURE 6: CHAT WITH MENTOR
Purpose: Send messages to mentor outside of structured sessions
User Flow:
1. Student navigates to Messages page
2. Existing conversations shown in left sidebar:
   ├─ Mentor avatar & name
   ├─ Last message preview
   ├─ Unread indicator
   └─ Time of last activity
3. Click conversation → opens message thread
4. Input field at bottom for new messages
5. Supports:
   ├─ Text messages
   ├─ Emoji reactions
   ├─ File attachments (PDF, images)
   └─ Link previews
6. Read receipts show when mentor reads message

Technical Details:
- GET /api/conversations
- GET /api/conversations/:id/messages
- POST /api/messages { conversationId, content, attachments }
- WebSocket for real-time message delivery
- Notifications for unread messages

STUDENT FEATURE 7: TRACK LEARNING PROGRESS
Purpose: Monitor goals and learning objectives in mentorship
User Flow (NOT IMPLEMENTED IN MVP):
1. On dashboard, student sees progress summary:
   ├─ Goals set (count)
   ├─ Sessions attended (count)
   ├─ Progress indicators
   ├─ Recent feedback
2. Goals page shows:
   ├─ List of personal learning goals
   ├─ Linked mentor connections
   ├─ Checkpoints (milestones)
   └─ Mentor notes on goal progress
3. Can update goal status or add notes

Technical Details:
- GET /api/student-progress
- POST /api/student-goals

4. MENTOR FEATURES (Alumni Providers)
================================================================================

MENTOR FEATURE 1: COMPLETE MENTOR PROFILE
Purpose: Alumni create detailed professional profile for discovery
User Flow:
1. After registration, alumni shown onboarding flow
2. Profile completion steps:
   └─ Step 1: Personal Information
      ├─ Profile picture (upload or avatar generator)
      ├─ Bio (rich text editor, 2000 char)
      ├─ Graduation year
      ├─ Current company
      └─ Industry (dropdown)
   └─ Step 2: Professional Details
      ├─ Expertise tags (multi-select, searchable)
      ├─ Years of experience
      ├─ LinkedIn URL (optional)
      ├─ Website/Portfolio (optional)
      └─ Certifications (optional multi-input)
   └─ Step 3: Mentoring Details
      ├─ Max mentees per cycle
      ├─ Preferred mentee background
      ├─ Mentoring style (short description)
      └─ Focus areas (what they mentor on)
3. Save and profile goes live

Technical Details:
- PATCH /api/alumni-profile/:id
- Rich text stored as JSON (Slate/ProseMirror format)
- Profile visibility: DRAFT → PUBLIC
- Image upload to CDN (Cloudinary/S3)

MENTOR FEATURE 2: MANAGE WEEKLY AVAILABILITY
Purpose: Set recurring open office hours when available for sessions
User Flow:
1. Mentor navigates to Availability/Calendar page
2. Weekly grid displayed (Mon-Sun, time slots like 2pm, 3pm, etc.)
3. Click time slot → toggle available/unavailable
4. Alternative: drag to create 1hr+ blocks
5. Can set exceptions (unavailable on specific dates)
6. Max sessions per week limit displayed (e.g., "5 of 5 slots full")
7. Preview shows "Will appear as available Tue 2-3pm, Wed 2-4pm, etc."
8. Publish/Save changes

Technical Details:
- POST/PATCH /api/availability
- Data: { mentorId, dayOfWeek, startTime, endTime, recurring: true }
- Exceptions stored separately: { date, isUnavailable: true }
- Availability recalculates with pending sessions

MENTOR FEATURE 3: REVIEW MENTORSHIP REQUESTS
Purpose: Review incoming requests from students and accept/decline
User Flow:
1. Mentor receives notification "New mentorship request from [Student]"
2. Navigates to Requests page, sees list:
   ├─ Student name, year, major, interests
   ├─ Request message (if provided)
   ├─ Goals alignment indicator
   ├─ "Accept" and "Decline" buttons
3. Click student name to view:
   ├─ Full profile summary
   ├─ Interactive goals alignment
   ├─ Previous mentorship history
4. Click "Accept":
   ├─ Optional message field ("Looking forward to working with you!")
   └─ Submit → mentorship active
5. Or click "Decline":
   ├─ Reason selector (not a good fit, too busy, specialization, etc.)
   └─ Optional message

Technical Details:
- GET /api/mentorship-requests?mentorId&status=pending
- PATCH /api/mentorship-requests/:id
  payload: { status: ACCEPTED|DECLINED, responseMessage }
- Triggers notification to student immediately
- Updates request count badge on navigation

MENTOR FEATURE 4: SCHEDULE AND HOST SESSIONS
Purpose: Coordinate session timing with mentees and host meetings
User Flow:
1. After accepting mentorship, student requests sessions
2. Mentor sees session request notification
3. Mentor navigates to Sessions page → "Pending" tab
4. Session request shows:
   ├─ Student name
   ├─ Current availability
   ├─ Proposed time (if student chose)
   ├─ Proposed topic
5. Can "Confirm" or "Propose Alternate Time"
6. Once confirmed, session appears in calendar
7. 30 min before session: "Join Session" button appears
8. Click to join video call (Zoom/Meet embedded)
9. Session timer displayed, can mark as complete when done

Technical Details:
- GET /api/sessions?mentorId&status=pending|scheduled
- PATCH /api/sessions/:id { status: ACCEPTED|DECLINED|ONGOING|COMPLETED }
- Video conferencing integration: create room on Zoom/Google Meet
- POST /api/sessions/:id/start → returns videoRoom.joinUrl

MENTOR FEATURE 5: RECEIVE AND MANAGE SESSION FEEDBACK
Purpose: View student feedback and ratings after sessions
User Flow:
1. Immediately after session, mentor gets notification:
   "Your session with [Student] on [date] has feedback"
2. Mentor can navigate to Feedback page:
   ├─ List of recent feedback
   ├─ Each feedback shows:
      ├─ Student name & date
      ├─ Rating (stars + avg)
      ├─ Student comment
      ├─ Focus areas mentioned
3. Can respond to feedback (optional acknowledgment)
4. Aggregate rating appears on mentor profile

Technical Details:
- GET /api/session-feedback?mentorId
- Mentor rating = average of all feedback.rating
- Displayed on profile: "⭐ 4.8 (32 reviews)"

MENTOR FEATURE 6: COMMUNICATE WITH MENTEES
Purpose: Direct messaging with mentees outside of sessions
User Flow:
1. From any mentorship, mentor can click "Message [Student]"
2. Opens conversation thread
3. Supports bidirectional messaging with all session mentees
4. Messages page shows:
   ├─ Active mentees (with open conversations)
   ├─ Archived mentees (past mentorships)
5. Can start conversation proactively

Technical Details:
- POST /api/conversations (if not exists)
- GET /api/conversations?mentorId
- WebSocket for real-time updates
- Background notification for unread messages

MENTOR FEATURE 7: MANAGE CURRENT MENTEES
Purpose: View and track all active mentee relationships
User Flow:
1. Mentor navigates to Mentees page
2. See grid or list of:
   ├─ Mentee name, photo, year, major
   ├─ Relationship status (Active, Completed, On Pause)
   ├─ Sessions completed count
   ├─ Last session date
   ├─ Next scheduled session (if any)
   └─ Quick actions: Message, Schedule Session, View Profile
3. Can filter by status or sort by last activity
4. Click mentee card → expanded profile with:
   ├─ Full mentee profile
   ├─ Shared notes
   ├─ Session history
   ├─ Feedback given
   └─ Goals progress

Technical Details:
- GET /api/mentees?mentorId&cycle=current
- Returns: student profile + relationship metadata
- Join query: MentorshipRequest + User + StudentProfile

MENTOR FEATURE 8: PROVIDE FEEDBACK TO MENTEES (OPTIONAL)
Purpose: Give constructive feedback on mentee progress
User Flow (NOT IMPLEMENTED IN MVP):
1. After session completion, mentor can leave feedback
2. Feedback form includes:
   ├─ Rating scale (how engaged was mentee, etc.)
   ├─ Comment field
   ├─ Progress indicators
3. Student can view all mentor feedback in progress summary

MENTOR FEATURE 9: VIEW MENTOR STATISTICS
Purpose: See personal mentoring performance and impact
User Flow (NOT IMPLEMENTED IN MVP):
1. Dashboard shows mentor stats:
   ├─ Total mentees (this cycle + all-time)
   ├─ Sessions hosted (count + avg duration)
   ├─ Average rating
   ├─ Most helped focus areas (from feedback)
   ├─ Top skills (that make students request them)

5. ADMIN FEATURES (Platform Management)
================================================================================

ADMIN FEATURE 1: CREATE AND MANAGE MENTORSHIP CYCLES
Purpose: Define temporal boundaries for mentor/student cohorts
User Flow:
1. Admin navigates to Cycles Management page
2. Current cycles displayed with status:
   ├─ Status badges (PLANNING, ACTIVE, COMPLETED, INACTIVE)
   ├─ Key dates
   ├─ Participant counts
3. Create new cycle:
   └─ Form fields:
      ├─ Year (2025)
      ├─ Season (select: SPRING/SUMMER/FALL/WINTER)
      ├─ Registration open date (date picker)
      ├─ Registration close date (date picker)
      ├─ Program start date
      ├─ Program end date
      ├─ Description/notes
      └─ Create button
4. Cycle created, auto-status PLANNING
5. Admin can update dates or mark ACTIVE when ready
6. Can archive past cycles

Technical Details:
- POST /api/cycles { year, season, dates, description }
- PATCH /api/cycles/:id { status, dates }
- Enrollment locked based on cycle dates
- Can only have ONE active cycle per year/season

ADMIN FEATURE 2: MODERATE USERS AND CONTENT
Purpose: Monitor user behavior and remove violators
User Flow:
1. Admin navigates to Users Management
2. Sees table/list of all users with:
   ├─ Name, email, role
   ├─ Account status (ACTIVE, SUSPENDED, DELETED)
   ├─ Report count (if flagged)
   ├─ Last activity date
   ├─ Actions: View, Ban, Suspend
3. Can search/filter by:
   ├─ Role (student, alumni, admin)
   ├─ Status
   ├─ Has reports (flagged/reported)
4. Click user → detailed profile view:
   ├─ Full profile information
   ├─ Associated sessions/feedback
   ├─ Report history
5. Click "Suspend" or "Ban":
   ├─ Duration picker (1 week, 1 month, permanent)
   ├─ Reason (required)
   ├─ Notify user checkbox
   └─ Execute button

Technical Details:
- GET /api/admin/users?page&limit&role&status&sort
- DELETE /api/admin/users/:id (hard delete - data anonymized)
- PATCH /api/admin/users/:id { status: SUSPENDED, reason, until }
- Email notification to user of suspension

ADMIN FEATURE 3: VIEW PLATFORM ANALYTICS & REPORTING
Purpose: Track system health, engagement, and key metrics
User Flow (NOT IMPLEMENTED IN MVP):
1. Admin Dashboard shows:
   ├─ Active users (students + mentors)
   ├─ Current cycle participant counts
   ├─ Total sessions this cycle
   ├─ Average session rating
   ├─ New sign-ups this week/month
   ├─ Top mentors by rating
   ├─ Feature usage charts (messages sent, requests, sessions, etc.)
2. Can drill down on:
   ├─ Cohort comparisons (Spring vs Summer participation)
   ├─ Mentor effectiveness (correlate ratings to mentee goals met)
   ├─ Churn metrics (mentees not attending sessions)

ADMIN FEATURE 4: SYSTEM CONFIGURATION
Purpose: Manage system settings and feature flags
User Flow (NOT IMPLEMENTED IN MVP):

6. MESSAGING & NOTIFICATIONS SYSTEM (SHARED)
================================================================================

REAL-TIME MESSAGING ARCHITECTURE:
- WebSocket connection established on user login
- Message delivery: HTTP POST (initial), WebSocket (real-time)
- Message status: SENDING → SENT → DELIVERED → READ
- Emoji reactions on messages
- File sharing support (max 25MB per file)
- Rich text/markdown support

NOTIFICATION TYPES:
1. REQUEST: New mentorship request received
2. REPLY: Mentor accepted/declined mentorship request
3. SESSION_INVITE: Student scheduled a session
4. SESSION_REMINDER: Session starts in 30 minutes
5. SESSION_FEEDBACK: Feedback received after session
6. MESSAGE: New direct message in conversation
7. SYSTEM: Admin announcements (cycle opened, etc.)

NOTIFICATION CHANNELS:
- In-app (page notification + badge count)
- Email (aggregate digest, 1x daily)
- Push notifications (if browser/mobile enabled)
- Can mute specific conversations

NOTIFICATION SETTINGS:
Users can control notification preferences:
├─ Delivery method (in-app, email, both)
├─ Frequency (real-time, digest, off)
├─ Mute specific users/conversations
└─ Do-not-disturb hours

7. FEATURE INTERACTIONS & WORKFLOWS
================================================================================

WORKFLOW 1: Complete Mentorship Cycle
┌─────────────────────────────────────────────┐
│ Admin creates cycle → PLANNING              │
│ Registration opens                          │
│                                             │
│ Students register & enter profile           │
│ Alumni register & set availability         │
│ Registration closes                         │
│                                             │
│ Cycle status → ACTIVE                       │
│ Students discover & send requests           │
│ Alumni review & accept/decline             │
│ Active mentorships begin                    │
│                                             │
│ Students attend sessions → leave feedback   │
│ Alumni provide mentoring → receive ratings  │
│                                             │
│ Cycle end date reached → COMPLETED         │
│ Relationships become read-only              │
│ Alumni can mentor next cycle                │
└─────────────────────────────────────────────┘

WORKFLOW 2: Student Finds and Engages Mentor
Student Profile Setup
    ↓
Discover Mentors (filter by skills/industry/availability)
    ↓
View Mentor Profile + Reviews
    ↓
Send Mentorship Request (with optional message)
    ↓
Wait for Response (& get notification)
    ↓
Request Accepted ──→ Request Declined (try another mentor)
    ↓
Mentorship Active
    ↓
Request Session (choose available time)
    ↓
Session Confirmed by Mentor
    ↓
Attend Session
    ↓
Leave Feedback & Rating
    ↓
Can Request Additional Sessions or End Mentorship

WORKFLOW 3: Alumni Receives and Manages Mentees
Alumni Profile Setup & Availability Set
    ↓
Configure Max Mentees
    ↓
Wait for Requests (notification)
    ↓
Review Incoming Requests
    ├─→ Accept Request → Mentorship Active → Schedule Sessions
    └─→ Decline Request
    ↓
Host Sessions
    ↓
Receive Feedback & Rating
    ↓
Manage Current Mentees List
    ↓
End-of-Cycle: Relationships Complete

JOURNEY 1: STUDENT FINDS FIRST MENTOR
────────────────────────────────────────
